# 中期检查报告

**题目**：面向分布式设备的Transformer模型层级与MCP工具组件的动态装载与调度

**日期**：2026年3月

---

## 一、选题描述

### 1.1 研究背景与问题

随着大规模语言模型（LLM）的快速发展，以Transformer为基础架构的模型参数量已从数亿增长至数千亿。在实际部署场景中，将这类模型运行于资源受限的边缘设备或分布式异构节点集群上，面临着显存容量、存储带宽与计算能力的多重约束。以本研究所使用的Qwen3-1.7B模型为例，其完整权重约占3.4GB存储空间，而单块香橙派昇腾开发板的可用NPU显存仅约1GB，无法一次性加载全部模型层级。

与此同时，现代大模型推理系统越来越多地引入工具调用能力（Tool Use），即模型在推理过程中可以动态调用外部工具（如天气查询、计算器、翻译服务等）以增强其能力边界。这类工具组件同样需要占用设备内存，且其调用时机具有明显的时间依赖性——工具调用只在模型生成特定格式的输出时才会触发，具有稀疏性和不可预测性。

上述两个问题共同构成了本研究的核心挑战：**在资源受限的分布式异构设备上，如何对Transformer模型层级与MCP工具组件进行动态装载与调度，以实现高效稳定的大模型推理？**

### 1.2 研究目标

本研究旨在构建一个面向分布式异构设备的Transformer层级与MCP工具组件动态装载与调度系统，具体目标包括：

1. **模型层级的分布式切分与流水线并行**：将Transformer模型按层切分到多个设备，通过流水线并行方式实现跨设备协同推理。
2. **Prefill/Decode双阶段按需加载**：针对推理的两个不同阶段（Prefill全序列处理与Decode逐token生成），分别编译和加载对应的模型，同一时刻每个节点只保留一组模型，节省设备内存。
3. **MCP工具组件的动态装载与调度**：实现工具的按需加载、LRU驱逐、跨设备调度与分布式执行，支持流式解析工具调用并异步执行。
4. **多轮对话与KV Cache管理**：在多轮对话场景下，通过KV Cache快照机制复用历史上下文，避免重复计算。

### 1.3 研究意义

本研究的成果可直接应用于边缘计算、物联网、移动端AI等资源受限场景，为大模型在端侧的高效部署提供系统性解决方案。同时，MCP（Model Context Protocol）作为近年来兴起的工具调用标准协议，将其与分布式推理系统深度融合，具有较强的前沿性与工程价值。

## 二、国内外研究进展综述

### 2.1 大模型分布式推理

大模型分布式推理是当前系统领域的研究热点。主流方法包括张量并行（Tensor Parallelism）、流水线并行（Pipeline Parallelism）和数据并行（Data Parallelism）三类。

**张量并行**方面，Megatron-LM（Shoeybi等，2019）提出了针对Transformer的列/行切分方案，将注意力头和FFN层的权重矩阵分布到多个GPU上，通过AllReduce通信同步梯度。这种方法通信开销较大，适合高带宽互联的同构集群。

**流水线并行**方面，GPipe（Huang等，2019）和PipeDream（Narayanan等，2019）将模型按层切分到不同设备，形成流水线。PipeDream进一步引入了1F1B（One Forward One Backward）调度策略，减少了流水线气泡。近年来，vLLM（Kwon等，2023）通过PagedAttention机制高效管理KV Cache，显著提升了推理吞吐量。

**边缘端分布式推理**方面，研究相对较少。llama.cpp支持CPU推理和量化，可在消费级硬件上运行中等规模模型。MLC-LLM（Chen等，2023）针对移动端和边缘设备进行了编译优化。然而，这些工作主要面向单设备场景，对多节点异构设备的协同推理支持有限。

### 2.2 模型动态加载与内存管理

在内存受限场景下的模型动态加载方面，FlexGen（Sheng等，2023）提出了一种面向单GPU的卸载策略，将模型权重、KV Cache和激活值在GPU、CPU和磁盘之间动态调度，以极低的硬件配置运行大模型。DejaVu（Liu等，2023）利用上下文稀疏性预测注意力头和FFN神经元的激活情况，实现动态剪枝以加速推理。

针对NPU等专用加速器，华为昇腾平台提供了ACL（Ascend Computing Language）编程接口，支持将ONNX模型编译为OM格式并在NPU上高效执行。本研究在此基础上实现了Prefill/Decode双模型的按需加载与切换机制。

### 2.3 工具调用与MCP协议

工具调用（Tool Use / Function Calling）能力最早由OpenAI在GPT-4中系统化引入，随后被Anthropic、Google等主流模型厂商广泛采用。在开源社区，Qwen、LLaMA等模型也通过特定的提示格式支持工具调用。

MCP（Model Context Protocol）是Anthropic于2024年提出的开放标准协议，旨在统一AI模型与外部工具、数据源之间的交互接口。MCP定义了工具发现（list_tools）、工具调用（call_tool）等标准接口，使得工具的开发与模型的使用解耦。目前，MCP已获得包括Claude、Cursor等主流AI产品的支持，生态正在快速扩展。

然而，现有工具调用研究大多聚焦于单机场景，对分布式设备上的工具调度问题关注较少。如何在多节点环境中高效地路由工具调用请求、管理工具组件的内存占用、实现工具执行与模型推理的并行，是本研究的创新点所在。

### 2.4 KV Cache管理

KV Cache是Transformer推理加速的关键技术。vLLM的PagedAttention将KV Cache组织为非连续的内存页，支持动态分配和共享。SnapKV（Li等，2024）通过压缩历史KV Cache减少内存占用。在多轮对话场景下，如何高效复用历史轮次的KV Cache是一个重要问题。本研究提出了基于快照（Snapshot）的KV Cache管理机制，支持在多轮对话中按需恢复历史上下文。

## 三、已经开展的研究工作

### 3.1 系统整体架构设计

本研究设计并实现了一个两层架构的分布式推理系统：

**底层**为基于ONNX Runtime的本地单机推理框架（`onnx_local`），用于在CPU上验证模型切分、Prefill/Decode双阶段推理、工具调用等核心逻辑的正确性。

**上层**为基于华为昇腾ACL的4节点分布式推理框架（`om_device`），部署于4块香橙派昇腾开发板（IP：192.168.137.100~103），实现真实硬件上的分布式推理与工具调度。

### 3.2 模型切分与ONNX导出

针对Qwen3-1.7B模型（28层Transformer，hidden_size=2048，num_heads=16，GQA kv_heads=8），实现了按层切分的ONNX导出脚本（`convert_to_onnx.py`）。模型被切分为以下组件：

- `embed.onnx`：词嵌入层
- `layers_0_6.onnx`：第0-6层Transformer Block（7层）
- `layers_7_13.onnx`：第7-13层Transformer Block（7层）
- `layers_14_20.onnx`：第14-20层Transformer Block（7层）
- `layers_21_27.onnx`：第21-27层Transformer Block（7层）
- `output.onnx`：RMSNorm + LM Head

每个组件分别导出Prefill版本（seq_len=512，无past KV输入）和Decode版本（seq_len=1，带past KV输入），共生成三组模型（system/prefill/decode），分别对应系统提示处理、用户输入处理和逐token生成三个阶段。

### 3.3 Prefill/Decode双模型按需加载机制

设计并实现了`_ensure_mode`机制，实现模型的按需加载与切换：

- 系统启动时不加载任何模型（`_loaded_mode = None`）
- 首次推理（Prefill阶段）时加载Prefill模型组
- 切换到Decode阶段时，卸载Prefill模型，加载Decode模型
- 后续Decode步骤不再切换，直到下一轮reset
- reset后，下一次Prefill再切换回来

这一机制使得每个节点在任意时刻只需保留一组模型（约原来内存的50%），有效解决了设备显存不足的问题。

### 3.4 4节点分布式流水线推理

在`om_device`框架中，实现了基于TCP Socket + Pickle序列化的节点间通信机制（`network.py`），定义了`DistributedMessage`消息格式，支持以下消息类型：

- `MSG_FORWARD`：前向传播数据（hidden_states）
- `MSG_RESULT`：生成结果（next_token）
- `MSG_RESET`：重置KV Cache（广播到所有节点）
- `MSG_SHUTDOWN`：关闭节点
- `MSG_TOOL_CALL`：工具调用请求（含target_device_id路由信息）
- `MSG_TOOL_RESULT`：工具执行结果（含request_id匹配信息）

推理流水线按照Node 0（头节点）→ Node 1（中间节点1）→ Node 2（中间节点2）→ Node 3（尾节点）→ Node 0的顺序单向流动，每个节点独立管理自己负责的7层KV Cache。

### 3.5 MCP工具调用系统

实现了完整的工具调用子系统，包含以下核心模块：

**StreamingToolCallParser（流式解析器）**：在模型生成过程中实时解析JSON格式的工具调用。采用括号计数法（bracket counting）检测完整的JSON对象，支持嵌套结构，并能正确处理Qwen模型的`<think>...</think>`思考块。解析器维护一个缓冲区，每次接收新生成的token文本后增量解析，检测到完整工具调用时立即触发执行，不阻塞后续生成。

**ToolManager（工具管理器）**：负责工具的注册、加载、执行与卸载，支持多设备管理。实现了基于使用频率加权的改进LRU驱逐策略：驱逐优先级分数 = 使用次数 / (当前时间 - 最后访问时间 + 1)，分数越低越优先被驱逐。所有操作均通过`threading.RLock`和设备级`threading.Lock`保证线程安全，工具执行支持超时控制（默认30秒）。

**ToolScheduler（工具调度器）**：实现了`Device0PreferredScheduler`调度策略，优先将工具调度到Node 0（头节点）执行，以减少跨设备数据传输开销。当Node 0负载超过阈值（500MB）时，选择负载最低的其他节点。调度器维护每个设备的负载状态和已加载工具集合。

**ToolCoordinator（工具协调器）**：整合解析、调度与执行，支持本地执行和远程执行两种模式。远程执行通过`DistributedMessage`消息系统路由到目标设备，使用UUID作为request_id进行请求-响应匹配。

**内置工具集**：实现了5个内置工具：
- `weather_tool`：天气查询（模拟API）
- `calculator_tool`：数学表达式计算（安全eval）
- `time_tool`：当前时间查询
- `translate_tool`：文本翻译
- `unit_converter_tool`：单位换算

### 3.6 多轮对话与KV Cache快照机制

在`onnx_local`框架中，设计并实现了基于KV Cache快照的多轮对话机制（详见`MULTITURN_DESIGN_V2.md`）：

- **base_kv_snapshot**：缓存工具描述（system prompt）处理后的KV状态，作为每轮对话的起点
- **question_kv_snapshot**：缓存当前问题处理后的KV状态，用于工具结果注入后的第二轮推理
- 每轮新问题开始时，从base_kv_snapshot恢复，避免重新处理工具描述
- 工具结果注入后，从question_kv_snapshot恢复，继续生成最终回答

## 四、已经取得的实验结果

### 4.1 模型切分与导出验证

成功将Qwen3-1.7B模型切分并导出为ONNX格式，生成了system/prefill/decode三组共21个ONNX文件（每组7个：embed + 4个block + output + config）。通过`compare_npy.py`和`compare_final.py`脚本对切分后的分布式推理结果与原始PyTorch模型输出进行了数值对比，验证了切分的正确性（最大绝对误差在浮点精度范围内）。

### 4.2 ONNX本地推理验证

在`onnx_local`框架下，成功实现了基于ONNX Runtime的本地推理，验证了以下功能：
- Prefill阶段（seq_len=512）与Decode阶段（seq_len=1）的正确切换
- KV Cache的正确写入与读取
- 多轮对话中历史上下文的正确复用
- 工具调用的完整流程（解析→调度→执行→结果注入→继续推理）

### 4.3 分布式推理框架搭建

完成了4节点分布式推理框架的代码实现，包括：
- 4个节点角色（HeadNode、MiddleNode×2、TailNode）的完整实现
- 节点间TCP通信的建立与消息传递
- 各节点OM模型转换脚本（`convert_to_om_node0.sh` ~ `convert_to_om_node3.sh`）
- 完整的部署文档与启动脚本

### 4.4 工具调用系统验证

在本地ONNX环境下，验证了以下工具调用场景：
- 单工具调用：模型生成工具调用JSON → 解析 → 执行 → 结果注入 → 继续生成
- 多工具并发调用：同一轮生成中触发多个工具，异步并发执行
- 工具调用失败处理：工具不存在或执行超时时的错误处理与回退

工具调用的端到端延迟（从检测到工具调用到结果注入完成）在本地CPU环境下约为50-200ms（取决于工具类型），不显著影响整体推理体验。

## 五、未来的进度计划

### 5.1 近期计划（第4-6周）

**硬件部署与联调**：将当前代码部署到4块香橙派昇腾开发板，完成OM模型转换（通过ATC工具将ONNX转换为昇腾NPU专用的OM格式），进行端到端的分布式推理联调，验证真实硬件上的功能正确性。

**性能基准测试**：建立系统性的性能评测体系，测量以下关键指标：
- 首token延迟（Time to First Token, TTFT）
- 每token生成延迟（Time Per Output Token, TPOT）
- 端到端吞吐量（tokens/s）
- Prefill/Decode模型切换开销
- 工具调用端到端延迟

### 5.2 中期计划（第7-10周）

**动态调度算法优化**：当前工具调度策略（Device0Preferred）较为简单，计划引入更智能的调度算法，综合考虑设备实时负载、工具历史调用频率、网络传输延迟等因素，实现自适应的工具调度决策。

**层级动态迁移**：探索在推理过程中动态迁移模型层级的可行性，即根据设备负载状态，将某些层从高负载节点迁移到低负载节点，实现更细粒度的负载均衡。

**KV Cache压缩**：研究在多轮对话场景下对历史KV Cache进行压缩的方法，以支持更长的对话历史而不超出设备内存限制。

### 5.3 后期计划（第11-14周）

**系统集成与优化**：将各模块进行系统集成，进行端到端的性能优化，包括通信协议优化（减少序列化开销）、内存管理优化（减少不必要的数据拷贝）等。

**对比实验**：设计对比实验，将本系统与以下基线方案进行比较：
- 单节点推理（无分布式）
- 静态加载（不使用按需加载）
- 无工具调用系统（纯文本推理）

**论文撰写**：整理实验数据，撰写毕业论文，重点阐述系统设计的创新点、实验结果的分析与讨论。

## 六、小结

本研究围绕"面向分布式设备的Transformer模型层级与MCP工具组件的动态装载与调度"这一主题，已完成了系统架构设计、模型切分与导出、Prefill/Decode双模型按需加载机制、4节点分布式流水线推理框架、MCP工具调用子系统（含流式解析、动态调度、分布式执行）以及多轮对话KV Cache管理等核心工作。在本地ONNX环境下验证了各模块的功能正确性，为后续在真实昇腾硬件上的部署与性能优化奠定了坚实基础。下一阶段的重点是完成硬件联调、建立性能评测体系，并在此基础上进行算法优化与系统集成。
